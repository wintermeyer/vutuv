## vutuv visual design system — "Direction A"

Every page must use this one visual language. It is "confident professional-bold":
deep heritage **blue** + a **coral** accent, white cards on a light grey canvas,
clean typography. **Dark mode follows the system** (`prefers-color-scheme`) — there
is **no theme toggle**, and every surface/text needs `dark:` variants.

### Two tracks — know which one you're touching

1. **Legacy controller+view pages** still use shared classes (`.card` / `.card-list`,
   `.editform` + inputs (`.editform__hint` is the muted helper line under a field, e.g.
   "Markdown is supported"; its `--ok` brand-blue / `--error` red variants color the live
   username-availability verdict that app.js writes under the new-username field),
   `.button` + variants, `.breadcrumbs`, `.profile-header` page
   title, `.pure-table`, `.alert`, `.tags`/`.badges`, `.imagebox`,
   `.profiles`, `.job`, `ol.tags`/`.upvote`, `ul.thumbs` (flex rows: capped
   thumbnail + truncating URL), `.card__tablewrap` (every in-card `<table>`
   sits in this overflow-x scroller so a wide table scrolls instead of being
   clipped by the card's `overflow:hidden` — `card_table_scroll_test.exs`
   enforces it), `.breakwrap` (break a long unbroken string and cap the line at a
   readable measure — use it instead of an inline `word-wrap`/`max-width` style),
   `.card__empty` empty-state line, `.error-page` 404/403/500
   card). They
   are styled **centrally** in `assets/css/components.css` — since the old vendored
   `legacy.css` was removed, that file is the **single source** for these classes,
   including element defaults (body canvas, h1/p/a, label, tables, dl) and the data-URI
   icons. **To restyle legacy pages, edit `components.css` — do NOT reskin
   per-template.** A new legacy page that reuses these classes gets the look for free.
   **Dark mode for all of it lives in the `@media (prefers-color-scheme: dark)` block
   at the end of `components.css`** — when you add a light rule with a hardcoded
   colour, add its dark counterpart there. `test/vutuv_web/dark_mode_css_test.exs`
   guards the canvas rule and that `legacy.css` stays deleted.
2. **New / hand-written pages** (the shell `ShellLive`, the LiveViews, `user/show.html.heex`,
   the logged-out auth/welcome pages via `<.auth_layout>`)
   use the **`VutuvWeb.UI` components** (see **Components** below) or, where no component
   fits, the **recipes** below. Prefer a component; fall back to a recipe. Reach for a
   green-field rewrite only when a page's UX needs rethinking (like the profile) —
   otherwise let track 1 handle it.

### Tokens (defined in `assets/css/app.css` `@theme`)

- **Brand blue:** `bg-brand-600` (primary), hover `bg-brand-700`; text `text-brand-700` / `text-brand-800`; tint `bg-brand-50`. Full scale `brand-50…900`.
- **Accent (unread counts / CTAs highlight):** `bg-accent` / `text-accent` (coral `#f97362`).
- **Surfaces:** page is grey (`slate-50`/`slate-100`); cards `bg-white`; borders `ring-slate-200`; text `text-slate-900 / 700 / 500`. **Muted/secondary text is `text-slate-600 dark:text-slate-400`** — passes WCAG AA on both white cards and the slate canvas. **Don't use bare `text-slate-400` as a text colour**: it fails AA (~2.6:1 on white) and is gone from the templates; `slate-400` now lives on only as the `dark:` half of the muted pair. Even `slate-500` is borderline directly on the slate-100 canvas (breadcrumbs, footer), so those use `slate-600`. A WCAG axe-core sweep keeps the key pages clean.
- **Dark:** `dark:bg-slate-900` (cards) / `slate-950` (page), `dark:ring-slate-800`, `dark:text-slate-100 / 300 / 400`. `app.css` sets `html { color-scheme: light dark }` and `root.html.heex` ships a `theme-color` meta per scheme (`#ffffff` / `#020617`).

### Canonical recipes (copy these exactly)

- **Page root:** `<main>` already gives `mx-auto max-w-6xl px-4`. A page's own wrapper is e.g. `py-6` (+ `grid gap-6 md:grid-cols-3` for a content+rail layout — `md`, not `lg`, so portrait iPads get the columns too). Don't add another `max-w`/`px` unless intentional.
- **Card:** `rounded-2xl bg-white p-6 shadow-sm ring-1 ring-slate-200 dark:bg-slate-900 dark:ring-slate-800` — or `<.card>`.
- **Section title:** `text-sm font-semibold uppercase tracking-wide text-slate-500` — or `<.section_title>`.
- **Primary button:** `rounded-lg bg-brand-600 px-4 py-2 text-sm font-semibold text-white hover:bg-brand-700`.
- **Secondary button:** `rounded-lg bg-slate-100 px-4 py-2 text-sm font-semibold text-slate-700 hover:bg-slate-200 dark:bg-slate-800 dark:text-slate-200 dark:hover:bg-slate-700`. The `dark:hover:` is required: without it the dark `slate-800` flips to the light `slate-200` on hover.
- **Text link / "Add" action:** `text-sm font-semibold text-brand-600 hover:text-brand-700`.
- **Input / select / textarea:** `w-full rounded-lg border border-slate-300 bg-white px-3 py-2 text-sm focus:border-brand-500 focus:outline-none dark:border-slate-700 dark:bg-slate-800 dark:text-slate-100`. This exact string is `VutuvWeb.UI.input_class/0` (imported everywhere) — call it in HEEx as `class={input_class()}` (compose with utilities via `class={[input_class(), "resize-y"]}`) on green-field forms (the post composer, the auth pages) instead of re-typing it.
- **Checkbox (consent/opt-in box):** `mt-0.5 h-4 w-4 rounded border-slate-300 text-brand-600 focus:ring-brand-500 dark:border-slate-600 dark:bg-slate-800` — the string is `VutuvWeb.UI.checkbox_class/0` (the companion to `input_class/0`); used by the sign-up form's consent checkboxes (email visibility, search engines, AI agents). Call it instead of re-typing.
- **Tag chip:** `inline-flex items-center gap-2 rounded-lg bg-brand-50 px-3 py-1.5 text-sm font-medium text-brand-700 dark:bg-brand-900/40 dark:text-brand-100` — or `<.chip>`.
- **Unread count badge:** small `rounded-full bg-accent px-1 text-[11px] font-bold text-white`; add `ring-2 ring-white dark:ring-slate-900` when it overlaps an icon or avatar.
- **Avatar:** `Vutuv.Avatar.display_url(user, :medium|:thumb)` in an `<img>`, `rounded-2xl` (tile) or `rounded-full` (list) + `object-cover`.
- **Link preview card** (profile Links section, main column): `grid gap-4 sm:grid-cols-2 sm:gap-x-18` (the `sm:gap-x-18` 72px desktop column gutter lines the two thumbnail columns up edge-for-edge with the Followers/Following cards below, whose content channel is `gap-6` + two `p-6` = 72px; row/mobile gap stays `gap-4`), each cell an `<a>` wrapping `<img src={Vutuv.Screenshot.url({url.screenshot, url}, :thumb)} width="400" height="264" loading="lazy" class="aspect-[400/264] w-full object-cover">` inside `overflow-hidden rounded-lg ring-1 ring-slate-200 dark:ring-slate-800`, then a `min-w-0` title/URL block (`truncate`). The screenshots come from `Vutuv.PageScreenshot` (headless Chromium, on link create/update); `Screenshot.url/2` falls back to `/images/screenshot.png`.

### Components (`VutuvWeb.UI` — imported everywhere; no explicit import needed)

`lib/vutuv_web/templates/user/show.html.heex` is the canonical example of the kit.

- `<.card class="…">…</.card>` — the card surface.
- `<.auth_layout title=… subtitle=…>…</.auth_layout>` — the logged-out **welcome/auth shell**: a brand-gradient hero panel (`from-brand-700 to-brand-500`, soft white decorative rings, white headline + `subtitle` + optional `:hero` slot for a member-count pill) beside a `<.card>` form (the default slot), stacking to one column under `md`. A `:headline` slot replaces the default `<h1>{title}` + subtitle when a hero needs full control over its heading typography (the landing page's founder-quote + attribution block); `title` stays required as the plain-text fallback. Used by the sign-up landing (`page/index`), `/login` and the two PIN screens so the logged-out entry flow matches the app instead of the old full-bleed photo "imagebox" (now deleted). The legal pages (impressum, datenschutz) are plain `<.page_header>` + `<.card>` content (no hero).
- `<.pin_actions />` — the two "stuck on the PIN page" escape hatches shared by both PIN-entry screens (login `session/pin_user_login` + post-registration `page/pin_new_registration`): a "Resend PIN" and a "Use a different email address" text-link, each a CSRF POST (to `~p"/login/resend"` / `~p"/login/cancel"`). Resend re-mails the PIN for the pending cookie identity (rate limited 5/hour); cancel drops the identity cookie so the landing page is no longer pinned to the PIN form. Uses the standard text-link recipe; no new tokens.
- `<.section_title>…</.section_title>` — uppercase muted heading inside a card.
- **Unified card UX (showcase + onboarding):** the profile is a **showcase**, not an editor. The dashed `<.empty_add>` tile is **onboarding scaffolding** — it shows only while a card is **empty** (so a brand-new owner sees an obvious way to start filling each section). Once a card has entries it is a **clean showcase** that reads like the visitor's view, with a quiet `<.card_footer_link>` "Verwalten ›" (→ the management page, where adding more / editing / deleting happens) or, for General Info, "Ändern ›" (→ `/edit`). There is no header "Add" button and no hidden ⋯ menu; existing entries are edited/removed with the calm `<.row_actions>` on the management pages. **`Beiträge` (Posts) is the one exception:** its compose tile stays pinned to the top of the card *always* (the Twitter/Facebook compose-on-top convention, since posting is the primary, frequent action).
- `<.section_header title={…}>` — card header row: section title + an optional right-aligned `:action` slot. Under the unified card UX the owner's add is the body `<.empty_add>` tile (empty cards only), not a header button, so most call sites pass just `title`; the `:action` slot stays for the rare non-add header control.
- `<.settings_nav user={…} active={:profile|:privacy|:notifications|:apps|:account} />` — the underline-tab sub-navigation shared by the **profile editor** (`/:slug/edit`, tab `:profile`) and the **settings pages** (`VutuvWeb.SettingsController`: `/:slug/settings/privacy`, `/:slug/settings/notifications`, `/:slug/settings/apps`, and `/:slug/settings` account hub `:account`). The old everything-on-one-page edit form was split into these focused, owner-only Track-2 pages; **each page's `h1` names its own tab** (Profile / Privacy / Notifications / Apps / Account), not a shared "Settings", so a deep-landing member can tell where they are (set the literal per template; do not derive it from the nav). Each settings/edit action also sets its own `:page_title` (`Edit profile` / `Privacy settings` / `Notification settings` / `Apps & API` / `Account settings`) so the browser `<title>` stops falling back to the bare member name. The nav is how a member moves between them. **What lives where** (clustered by what a human comes to do): **Profile** = public content only (photos, name, about you); **Privacy** = a *Search engines & AI* card (the search-engine/AI consent **checkboxes** — `checkbox_class()`, reusing the sign-up form's inverted `checked_value: "false"` trick for the opt-out `noindex?`/`noai?` so the box reads positively, "Allow …"; helper line forced `font-normal` because the wrapping `<label>` defaults to bold in `components.css`) **plus** a *Safety* card with two rows, `/blocks` (blocked members) and `/moderation/cases` (content under review) — both moved here from the account hub, where members did not look for them; **Notifications** = an *Email notifications* card with one positive-flag checkbox **per type** (`notification_emails?` unread messages — default on; `email_on_connection_request?` / `email_on_endorsement?` / `email_on_follower?` — opt-in, default **off**), each wired to a real Emailer notice (see the notification-email note below), **plus** a read-only *In-app notifications* explainer linking to `/notifications`; **Apps** = the *Apps & API access* card (connected apps + access tokens + a "Read the API docs" cross-link to `/developers`), split off onto its own tab; **Account** = *Sign-in & identity* (username, emails), *Language & region* (the interface-language `locale` select — a small form posting `PUT|PATCH /:slug/settings/language` → `update_language`, kept off the profile editor since it sets the whole UI, not the public profile), *Your data* (export), and the red *Delete account* danger card. The dissolved "Privacy & security" grab-bag card is gone. Account-hub / Apps / Safety rows render through `<.account_row>` (in `SettingsHTML`). The ad banner is suppressed on every `/:slug/edit` and `/:slug/settings/*` page (`VutuvWeb.Plug.AdBanner`).
- `<.card_menu id="…"><:item href={…}>…</:item></.card_menu>` — **Track 2** per-card ⋯ dropdown (native `<details data-menu>`; `app.js` closes it on outside click and Escape). **No longer used on the profile sections** — they switched to the dashed `<.empty_add>` tile + `<.manage_footer>` (the quiet ⋯ failed non-technical owners; see the unified card UX note above). Its one remaining home is the **post card's author menu** (`VutuvWeb.PostComponents`): Edit + a confirmed red Delete (`:item` supports `method`, `confirm` data-confirm and `danger` red item). The owner guard sits on the slot at the call site.
- `<.empty_add href={~p"/…/new"} class={…}>{gettext("Add …")}</.empty_add>` — the owner's **add tile** / onboarding scaffold: a full-width dashed-outline tile (plus glyph + label, brand-tint hover, `data-empty-add` hook) linking to the new-entry form. On the **profile** it shows only while the card is **empty** (`:if={same_user?(…) and <collection empty>}`); once there are entries the card is a clean showcase with a `<.card_footer_link>` "Verwalten ›" instead. The label is a clear per-section call to action (`gettext("Add a link")`, `Add work experience`, `Add a tag` → `/tags/new`). The two exceptions: **Beiträge** keeps its compose tile always (`Write your first post` empty / `Write a post` populated → `/feed`); **General Info** has no list, so its empty tile (→ `/edit`) graduates to an "Ändern ›" footer once set. On the **management pages** `<.card_section>` still renders the tile above the list (empty *or* populated), since those are the editor; a **visitor's** empty management index shows the plain `card__empty` line instead.
- `<.row_actions edit_to=… delete_to=… confirm=… class=… />` — the calm, labeled per-entry **Edit/Delete** actions on every management list and entry show page: a right-aligned brand "Edit" text link + a muted-red "Delete" text link that deletes through the `delete` method (CSRF) behind a `data-confirm` prompt (defaults to "Are you sure?"). The unified replacement for the loud `<.edit_delete_actions>` pencil + red-trash-circle pair on user-facing pages; omit `edit_to` for delete-only rows (tags). Keep the owner guard at the call site (`:if={same_user?(…)}`).
- `<.card_footer_link href={~p"/…"}>{…}</.card_footer_link>` — the quiet card **footer link** for "see more / manage" navigation below a card's entries (the profile's Posts/Followers/Following "View all" links and, via `<.manage_footer>`, the section cards). A **centered, muted** text link (slate → brand on hover) with a hairline top divider and a trailing chevron, so it reads as a calm footer subordinate to the prominent dashed `<.empty_add>` tile up top — pairing the styled tile with a bare brand link looked like two design vocabularies. Guard rendering with `:if` at the call site.
- `<.manage_footer href={~p"/…"} total={…} preview={…} owner={…} />` — the owner's visible bridge from a profile section card to its management page, folded with the public "View all" link: a `<.card_footer_link>` reading "View All (N)" once `total > preview` (everyone), else "Manage" for the owner when `total >= 1`, else nothing. Replaced the old "View All" links + the ⋯ menu's "Manage entries" so removing the menu doesn't strand the owner with no way to edit existing entries.
- `<.button>` — `primary` (default) or `variant="secondary|ghost|danger"`. Renders a link when given `navigate`/`patch`/`href` (add `method="post|delete"` for CSRF actions), else a `<button>` (set `type="submit"`). e.g. `<.button href={~p"/follows?#{…}"} method="post">Follow</.button>`.
- `<.follow_button variant="icon|text|button" follower_id={…} followee_id={…} follow_id={…} />` — the **single owner of the two `~p"/follows…"` route shapes** (CSRF DELETE `/follows/<id>` to unfollow, POST `/follows?follow[…]` to follow, branched on `follow_id` — `nil`/`false` = not following). This is the **follow** (one-directional subscription); the separate mutual **connection** request/accept control lives on the profile header, not here. The following-state lookup (`following_by_id` map or `user_follows_user?/2`) stays at the call site; the component just consumes its result. Three visual variants reproduce the legacy follow controls exactly: `icon` = `button button--icon` + `i.icon.icon--follow|--unfollow` glyphs (the `card_list` rows), `text` = the muted/brand text link reading "Following"/"Follow" (the `user_row` rail), `button` = a secondary/primary `<.button>` "Following"/"Follow" (the profile header + the post-teaser, which renders only the follow half — pass `follow_id={nil}`). `icon`/`text` render through `button/2` (a `<button>` with `data-method`); `button` renders an `<a>` via `<.button>`. Keep the owner / visitor / logged-in guard on a `:if` at the call site.
- `<.connect_control state={…} target_id={…} connection_id={…} />` — the **mutual connection** control on the profile header (the counterpart to `<.follow_button>`, which is the one-directional follow). Owns the `~p"/connections…"` route shapes and branches on the `state` atom from `Vutuv.Social.connection_state/2`: `:none`/`:declined` → primary **Connect** (POST `/connections`, `connection[user_id]`), `:pending_sent` → secondary **Pending** that withdraws (DELETE `/connections/<id>`), `:pending_received` → primary **Accept** (POST `/connections/<id>/accept`) + secondary **Ignore** (POST `/connections/<id>/decline`), `:accepted` → a disabled secondary **✓ Connected** (disconnect lives on the connections page). Keep the visitor / logged-in guard on a `:if` at the call site.
- `<.avatar user={@user} size="xs|sm|md|lg" shape="circle|square" />` — resolves `Vutuv.Avatar`; or pass `src=`. A user **without a picture** renders an **initials tile** (brand-tint `<span>`, initials via the public `name_initials/1`, same module — also used by the shell's top-bar tile); only a call with neither `user` nor `src` gets the neutral SVG placeholder.
- `<.chip>` — tag chip; pass `navigate`/`href` to render it as a link.
- `<.secret_once flash={@flash} key={:new_token} label={…} class="mt-4" />` — the **show-once credential reveal** (access tokens, OAuth client secrets, webhook signing secrets): a brand-tint box with a `select-all` `<code>` line, rendered only while the one-shot flash under `key` holds the freshly minted secret (`data-secret-once` hook). Every "copy it now, it will not be shown again" box goes through this; don't hand-roll a fourth.
- `<.count_badge count={n} class="…" />` — coral unread badge; renders nothing when 0; `class` positions it. Add `ring-2 ring-white dark:ring-slate-900` via `class` when it overlaps an icon/avatar (the shell does this).
- `<.frozen_banner class="…">{text}</.frozen_banner>` — the owner-facing **moderation freezer notice** (amber strip, ⚑ glyph, trailing "Review" link to `/moderation/cases`, `data-frozen-banner` hook). The one rendering of "only you can see this while a report is handled"; the post card passes `class="mb-3 rounded-lg px-3 py-2 text-xs"`, the profile header `class="rounded-2xl px-4 py-3 text-sm"`. Guard with `:if={owner and frozen}` at the call site. Amber (`amber-50/800`, dark `amber-900/30`/`amber-200`) is reserved for moderation notices.
- `compact_count/1` — **every rendered count goes through this** (plain function, same module): exact up to 999, above that floored to `1K` / `80K` / `5M`. `<.count_badge>` applies it already; call it yourself for inline numbers (follower counts, "Load N of M more"). Page-number links in `<.pager>` are navigation, not counts — they stay exact.
- `highlight/2` — wraps every case-insensitive occurrence of the needle(s) in a text in a brand-tinted `<mark>` (`bg-brand-100 text-brand-900`, dark `bg-brand-500/30 text-brand-100`): the search result match marker (people names, tag chips, post excerpts on `/search`). Plain function, same module; returns safe HTML from escaped parts. Phonetic ("similar names") results deliberately get no marker — the absent highlight is part of the exact-vs-similar distinction.
- `delimited_count/1` — the **exact**, thousands-grouped form (`60123` → `"60,123"`, German `"60.123"`), the deliberate exception to `compact_count/1` for the **landing-page member counter**, which shows the precise total and ticks it up live. Rendered by the embedded `VutuvWeb.MemberCountLive` pill (fed by `Vutuv.Accounts.MemberCounter`), not the controller.
- `<.page_header title={…} crumbs={[…]} />` — **legacy Track 1** page top: the `.profile-header > .profile-header__info > h1` block (rendered when `title` is set) and/or the `.breadcrumbs` row wrapping `gen_breadcrumbs/1` (rendered when `crumbs` is set). `title` is a plain string the call site already builds (e.g. `gettext("Emails belonging to ") <> full_name(@user)` or `gettext("Tags of %{name}", name: full_name(@user))`); `crumbs` is the list you used to hand to `gen_breadcrumbs/1` (`gen_breadcrumbs/1` is called fully qualified inside the component because `VutuvWeb.UI` doesn't import `UserHelpers`). new/edit pages pass only `crumbs`. Replaces the ~47 header + ~65 breadcrumb boilerplate blocks across the controller pages. Pages whose header carries more than the single h1 (avatar, buttons, …) keep their hand-written markup; `user/show.html.heex`'s bespoke profile header is excluded.
- `<.form_error changeset={@changeset} />` — **legacy Track 1** changeset-error banner (`.alert.alert-danger` + `.editform__error`, styled by `components.css`, not utilities); renders only when `@changeset.action` is set. Shared by the `editform` `form_content` templates so the banner is written once.
- `<.form_actions backlink={@backlink} delete_to={assigns[:delete_to]} confirm=… />` — **legacy Track 1** Cancel/Submit actions row (`.editform__actions` with a `.button.button--cancel` Cancel link to `@backlink` and a `.button` Submit). Shared by the `editform` `form_content` templates. On **edit** forms, `delete_to` appends the canonical delete control (`id="delete-entry"`, a `.button--danger` link, CSRF-protected DELETE behind `data-confirm`; `confirm` overrides the default "Are you sure?"). The shared `form_content` templates thread it through as `delete_to={assigns[:delete_to]}` and only the `edit.html.heex` wrappers pass it, so new-forms stay delete-free — `test/vutuv_web/controllers/profile_edit_affordances_test.exs` enforces both directions. Forms with a custom submit label or no Cancel keep their own hand-written row.
- `<.edit_delete_actions edit_to=… delete_to=… show_to=… confirm=… class=… />` — **legacy** icon-button group: the `.btns-right` wrapper + `.button.button--icon.button--small` CSS-glyph icons (`i.icon.icon--edit|--delete|--search`), view → edit → delete order, delete via the `delete` method (CSRF) + `button--danger`. **Now used only by the admin/group index tables** — the user-facing profile management lists/shows switched to the calm `<.row_actions>` (see above). Each `*_to` is optional; `title_*` set tooltips, `:extra` adds a trailing button. Keep the owner/admin guard at the call site.
- `<.card_section empty={…} add_href={…} add_label={…} empty_text={…}>…</.card_section>` — **legacy Track 1** card shell (`<div class="card-list"><section class="card">…</section></div>`, styled by `components.css`). Under the unified card UX its "Add" is the dashed `<.empty_add>` tile (label from `add_label`) rendered **above** the content (compose-on-top), the same way whether the card is empty or populated (the old bottom `.card__morelink` and the header add button are both gone). A visitor (no `add_href`) on an empty card still gets the plain `<p class="card__empty">` line (`empty_text`, default gettext("Nothing here yet.")). New/edit form wrappers pass no `add_href`, so they just wrap the inner block. Used by the email/phone_number/url/social_media_account/work_experience/address/user_tag/group/search_term indexes + the new/edit wrappers.
- `<.pager params={@conn.params} total={@row_count} />` — numbered pagination for offset-paginated **browse** pages (followers, tags, users); the math lives in `Vutuv.Pages` (`paginate/3` on the query, 250/page). Renders nothing when one page fits. **Feed** LiveViews (notifications, /feed posts) paginate with a cursor + "Load more" button appended into the stream instead (`Vutuv.Activity.notifications_page/2`, `Vutuv.Posts.feed_page/2`) — don't mix the two styles.
- `<.load_more :if={@more?} />` — the cursor-pagination "Load more" control those feed-style LiveViews share: a centered secondary `<.button id="load-more">` that emits the `"load-more"` event. The inner block overrides the default label (notifications shows the remaining count); `class` adds wrapper utilities (e.g. `mt-6`).
- `<.icon_repost class=… />` / `<.icon_reply class=… />` / `<.icon_bookmark filled?={…} class=… />` — the three Track 2 outline SVG icons used in more than one module (post card "Reposted by"/"Replying to" lines + action bar; shell + action bar). Size via `class` (defaults `h-5 w-5`); `filled?` switches the bookmark to solid. Icons used by a single LiveView stay private to it by convention.
- `<.post_card post={…} viewer={…} mode={:preview|:full} surface={:card|:flat}>` — **Track 2** post card, the one rendering of a post everywhere (feed + profile "Posts" section use `:preview`, the permalink page `:full`). `surface={:card}` (default) stands alone; `surface={:flat}` renders without the card shell for embedding inside an existing card — the profile's Posts section stacks flat posts in a `divide-y divide-slate-100 dark:divide-slate-800` list with `py-4 first:pt-0 last:pb-0` rows, under a `<.section_header>` whose owner `add_href` points at the `/feed` composer. Author header (avatar/name/time→permalink, "edited" hint, generic 🔒 for restricted posts — never the deny list), markdown body in `.markdown` (preview adds `line-clamp-5` + server-side block-boundary cut with a "Read more" link), preview thumbnail row vs full gallery (inline-referenced attachments render in place, the rest below), tag chips. When `viewer` is the author, the card renders its own top-right `<.card_menu id={"post-menu-#{post.id}"}>` with Edit and a confirmed red Delete — no slot, every rendering of a post carries it. Lives in `VutuvWeb.PostComponents` and is **not** globally imported — `import VutuvWeb.PostComponents` at the call site.
- **Markdown body recipe:** wrap rendered Markdown in `<div class="markdown text-slate-800 dark:text-slate-200">`. `.markdown` (in `components.css`, light + dark) carries only structure — paragraph/list/blockquote/code spacing, tables (`display:block` + `overflow-x:auto` so a wide table scrolls inside the card, hairline row borders; used by the `/developers` docs) and `img.post-inline-image` — text color comes from the wrapper's utilities. **User-written** Markdown (posts) renders through `VutuvWeb.Markdown.render_post/2` (inline images restricted to own attachments) / `render_preview/2`, never raw Earmark; repo-authored trusted content (`priv/dev_docs` via `DevDocController`) may use Earmark directly.

### Shell & layout facts (don't re-implement)

- The chrome — sticky top bar + mobile bottom tab bar with the live unread badges — is `VutuvWeb.ShellLive`, embedded in `app.html.heex`. Pages render **inside** it; never add their own nav.
- **Flash = top-right toasts** (`#toast-tray` in `app.html.heex`). Never add inline flash banners; `VutuvWeb.LayoutHTML.flash/1` and its empty partial were deleted along with the last inline calls.
- In-app real-time events go through `Vutuv.Activity` (PubSub on `"user:<id>"`) and `VutuvWeb.Presence`.
- **Gettext locale must be set per process.** `VutuvWeb.Plug.Locale` resolves it per
  request and stores it in the session; LiveViews re-apply it on mount via
  `VutuvWeb.LiveLocale` (called from `Live.InitAssigns` and `ShellLive.mount`).
  A new LiveView mounted outside the `live_session` must call it too, or its copy
  (and the whole shared chrome) silently falls back to English.
- **Legacy page anatomy:** `.profile-header` h1 → `.breadcrumbs` → `.card-list` >
  `.card` — the header + breadcrumbs are emitted by `<.page_header title=… crumbs=… />`
  (new/edit pages pass only `crumbs`); the owner "Add" is the dashed `<.empty_add>`
  tile that `<.card_section add_href=…>` renders above the list (empty *or* populated),
  per-entry edit/delete is the calm `<.row_actions>` (placed in a `td.text-right` for
  table rows), and a visitor's empty collection renders `<p class="card__empty">`.
  Copy this from `email/index`. The admin/group index tables still use the legacy
  `<.edit_delete_actions>` icon group.
- Error pages (`VutuvWeb.ErrorHTML`) render the `.error-page` card (code, message,
  "Back to the start page") and must work with and without the app layout.

### CSS architecture (don't break it)

`app.css` order: `@import "tailwindcss"` → `@import "./components.css" layer(components)`,
**at the top** before `@source`/`@theme` — the `@import … layer()` only works there
(CSS ignores `@import` after other rules). `components.css` lives in the `components`
layer, so it beats Preflight (base) but loses to Tailwind utilities, which lets
hand-written pages use utilities freely. There is **no `legacy.css` anymore** — do not
reintroduce it (a regression test enforces this). Dev serves `/assets/app.css`
undigested, so hard-reload (Cmd+Shift+R) after a rebuild. **Dev does not use
`tailwind --watch`:** the v4 CLI's watch mode rebuilds from a cached copy of
`@import`ed CSS and ignores CSS edits outright (verified on 4.0.0 and 4.3.0), so
`VutuvWeb.TailwindWatcher` (wired as the dev watcher) runs a correct one-shot
build on every CSS/template/JS change instead. Edits to `components.css` reach
the browser within ~2s; if they ever don't, suspect that watcher first.

### Don'ts

- No theme toggle (dark = system).
- Don't reskin legacy pages per-template — edit `components.css`.
- Don't invent new bespoke colours; stay on brand / accent / slate.
- No em-dashes in UI copy (project-wide rule).
