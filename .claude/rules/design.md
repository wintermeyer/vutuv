## vutuv visual design system ŌĆö "Direction A"

Every page must use this one visual language. It is "confident professional-bold":
deep heritage **blue** + a **coral** accent, white cards on a light grey canvas,
clean typography. **Dark mode follows the system** (`prefers-color-scheme`) ŌĆö there
is **no theme toggle**, and every surface/text needs `dark:` variants.

### Two page styles ŌĆö know which one you're touching

Every page is styled one of two ways: a **classic page** (legacy controller +
`.html.heex` view page, styled by the shared classes in `assets/css/components.css`)
or a **kit page** (hand-written markup built from the `VutuvWeb.UI` component kit +
Tailwind recipes). This is a *styling* distinction, not a module category ŌĆö a
LiveView can sit on either style.

1. **Classic pages** (legacy controller+view pages) still use shared classes (`.card` / `.card-list`,
   `.editform` + inputs (`.editform__hint` is the muted helper line under a field, e.g.
   "Markdown is supported"; its `--ok` brand-blue / `--error` red variants color the live
   username-availability verdict that app.js writes under the new-username field),
   `.button` + variants, `.breadcrumbs`, `.profile-header` page
   title, `.pure-table`, `.alert`, `.tags`/`.badges`, `.imagebox`,
   `.profiles` (+ the `.profiles--tags` variant whose rows grow to fit a
   `.profile-tags` line: up to 4 small tag links plus a `.profile-tags__more`
   "+N more tags" overflow count, used by the most-followed listing via
   `card_list`), `ol.tags`/`.upvote`, `ul.thumbs` (flex rows: capped
   thumbnail + truncating URL), `.reorder*` (the owner's drag-and-drop ordering
   tool shared by every orderable section page ŌĆö links, phone numbers, addresses,
   social media accounts, emails: `.reorder__item` draggable tile =
   `.reorder__handle` grip + `.reorder__body` (section-specific, with a
   `.reorder__thumb` for links) + `.reorder__move` up/down `.reorder__btn` arrows
   (sized as touch targets) + `<.row_actions>`. It is **not** a classic page: it is
   the embedded `VutuvWeb.SectionReorderLive` (`live_render`, owner-only, like the
   shell), so both drag and arrows persist over the socket with no reload ŌĆö the
   `Reorder` hook in `app.js` does the drag, the arrows are `phx-click`. The hook
   also FLIP-animates the arrow reorders (snapshot row tops in `beforeUpdate`,
   glide them from old to new spot in `updated`, ~180ms; honors
   prefers-reduced-motion). Mobile-first: touch can't fire native drag, so the
   arrows are the reorder path on phones), `.card__tablewrap` (every in-card `<table>`
   sits in this overflow-x scroller so a wide table scrolls instead of being
   clipped by the card's `overflow:hidden` ŌĆö `card_table_scroll_test.exs`
   enforces it), `.breakwrap` (break a long unbroken string and cap the line at a
   readable measure ŌĆö use it instead of an inline `word-wrap`/`max-width` style),
   `.card__empty` empty-state line, `.card__label` (the small uppercase muted
   caption for field/section labels inside a legacy card ŌĆö e.g. the detail show
   pages and the tag card; it is `<h2 class="card__label">` so each page keeps
   exactly one `<h1>` while the prominent `.card h2` styling stays available),
   `.error-page` 404/403/500
   card). They
   are styled **centrally** in `assets/css/components.css` ŌĆö since the old vendored
   `legacy.css` was removed, that file is the **single source** for these classes,
   including element defaults (body canvas, h1/p/a, label, tables, dl) and the data-URI
   icons. **To restyle legacy pages, edit `components.css` ŌĆö do NOT reskin
   per-template.** A new legacy page that reuses these classes gets the look for free.
   **Dark mode for all of it lives in the `@media (prefers-color-scheme: dark)` block
   at the end of `components.css`** ŌĆö when you add a light rule with a hardcoded
   colour, add its dark counterpart there. `test/vutuv_web/dark_mode_css_test.exs`
   guards the canvas rule and that `legacy.css` stays deleted.
2. **Kit pages** (new / hand-written: the shell `ShellLive`, the LiveViews,
   `user/show.html.heex`, the logged-out auth/welcome pages via `<.auth_layout>`)
   use the **`VutuvWeb.UI` components** (see **Components** below) or, where no component
   fits, the **recipes** below. Prefer a component; fall back to a recipe. Reach for a
   green-field rewrite only when a page's UX needs rethinking (like the profile) ŌĆö
   otherwise keep it a classic page.

### Tokens (defined in `assets/css/app.css` `@theme`)

- **Brand blue:** `bg-brand-600` (primary), hover `bg-brand-700`; text `text-brand-700` / `text-brand-800`; tint `bg-brand-50`. The scale is **sparse** ŌĆö only these steps exist in `@theme`: `50, 100, 200, 300, 400, 500, 600, 700, 800, 900` (**no `950`**). `brand-300` (`#93c5fd`, the natural light-blue step used for dark-mode link/hover text) was added once ~14 files relied on it and it was silently compiling to nothing; `brand-950` still doesn't exist, so a `brand-950` class silently compiles to nothing ŌĆö pick a defined step. After adding/changing utility classes in a `.ex`/`.heex`, rebuild (`mix tailwind vutuv`) and confirm each shipped by grepping `priv/static/assets/app.css` for the **escaped** selector (variants escape `:` to `\:`, e.g. `grep -F 'dark\:hover\:bg-brand-900'`).
- **Accent (unread counts / CTAs highlight):** `bg-accent` / `text-accent` (coral `#f97362`).
- **Surfaces:** page is grey (`slate-50`/`slate-100`); cards `bg-white`; borders `ring-slate-200`; text `text-slate-900 / 700 / 500`. **Muted/secondary text is `text-slate-600 dark:text-slate-400`** ŌĆö passes WCAG AA on both white cards and the slate canvas. **Don't use bare `text-slate-400` as a text colour**: it fails AA (~2.6:1 on white) and is gone from the templates; `slate-400` now lives on only as the `dark:` half of the muted pair. Even `slate-500` is borderline directly on the slate-100 canvas (breadcrumbs, footer), so those use `slate-600`. A WCAG axe-core sweep keeps the key pages clean.
- **Dark:** `dark:bg-slate-900` (cards) / `slate-950` (page), `dark:ring-slate-800`, `dark:text-slate-100 / 300 / 400`. `app.css` sets `html { color-scheme: light dark }` and `root.html.heex` ships a `theme-color` meta per scheme (`#ffffff` / `#020617`).

### Canonical recipes (copy these exactly)

- **Page root:** `<main>` already gives `mx-auto max-w-6xl px-4`. A page's own wrapper is e.g. `py-6` (+ `grid gap-6 md:grid-cols-3` for a content+rail layout ŌĆö `md`, not `lg`, so portrait iPads get the columns too). Don't add another `max-w`/`px` unless intentional. **Give the main content column `min-w-0`** (e.g. `min-w-0 space-y-4 md:col-span-2`): Tailwind's `md:grid-cols-*` compiles to safe `minmax(0,1fr)` tracks, but only at `md+`; below `md` (every phone) the grid falls back to an implicit `auto` track that respects the item's min-content, so a single `truncate`/unbreakable descendant (a threaded reply's parent-excerpt, a markdown body) forces the column ŌĆö and the whole page ŌĆö wider than the viewport and scrolls it sideways. `min-w-0` lets the column shrink below that min-content. `mobile_overflow_test.exs` guards the feed / profile / dev-doc columns.
- **Card:** `rounded-2xl bg-white p-6 shadow-sm ring-1 ring-slate-200 dark:bg-slate-900 dark:ring-slate-800` ŌĆö or `<.card>`.
- **Section title:** `text-sm font-semibold uppercase tracking-wide text-slate-500` ŌĆö or `<.section_title>`.
- **Primary button:** `rounded-lg bg-brand-600 px-4 py-2 text-sm font-semibold text-white hover:bg-brand-700`.
- **Secondary button:** `rounded-lg bg-slate-100 px-4 py-2 text-sm font-semibold text-slate-700 hover:bg-slate-200 dark:bg-slate-800 dark:text-slate-200 dark:hover:bg-slate-700`. The `dark:hover:` is required: without it the dark `slate-800` flips to the light `slate-200` on hover.
- **Text link / "Add" action:** `text-sm font-semibold text-brand-600 hover:text-brand-700`.
- **Input / select / textarea:** `w-full rounded-lg border border-slate-300 bg-white px-3 py-2 text-sm focus:border-brand-500 focus:outline-none dark:border-slate-700 dark:bg-slate-800 dark:text-slate-100`. This exact string is `VutuvWeb.UI.input_class/0` (imported everywhere) ŌĆö call it in HEEx as `class={input_class()}` (compose with utilities via `class={[input_class(), "resize-y"]}`) on green-field forms (the post composer, the auth pages) instead of re-typing it. **On a field that can fail validation, call `input_class(form, field)` instead**: same recipe, but once the field carries an error the slate border swaps for red (`border-red-400 focus:border-red-500 dark:border-red-500/70`), keeping the `<.form_error>` banner's "fields marked in red" true on kit pages (classic editform pages get the marking from `.editform__field--error`). Pair it with `"aria-invalid": f.errors[:field] && "true"` on the input and `error_tag f, :field` below it; when the field also has a muted hint line, render the hint only while there is **no** error (`:if={!f.errors[:field]}`) so one clear line speaks, not a stack (the sign-up tags field is the canonical example). A hand-built form with no changeset (the admin preference forms) uses `input_class(invalid?)` with a bare boolean ŌĆö same two strings.
- **Checkbox (consent/opt-in box):** `mt-0.5 h-4 w-4 rounded border-slate-300 text-brand-600 focus:ring-brand-500 dark:border-slate-600 dark:bg-slate-800` ŌĆö the string is `VutuvWeb.UI.checkbox_class/0` (the companion to `input_class/0`); used by the sign-up form's consent checkboxes (email visibility, search engines, AI agents). Call it instead of re-typing.
- **Tag chip:** `inline-flex items-center gap-2 rounded-lg bg-brand-50 px-3 py-1.5 text-sm font-medium text-brand-700 dark:bg-brand-900/40 dark:text-brand-100` ŌĆö or `<.chip>`.
- **Unread count badge:** small `rounded-full bg-accent px-1 text-[11px] font-bold text-white`; add `ring-2 ring-white dark:ring-slate-900` when it overlaps an icon or avatar.
- **Avatar:** `Vutuv.Avatar.display_url(user, :medium|:thumb)` in an `<img>`, `rounded-2xl` (tile) or `rounded-full` (list) + `object-cover`.
- **Link preview card** (profile Links section, main column): `grid gap-4 sm:grid-cols-2 sm:gap-x-18` (the `sm:gap-x-18` 72px desktop column gutter lines the two thumbnail columns up edge-for-edge with the Followers/Following cards below, whose content channel is `gap-6` + two `p-6` = 72px; row/mobile gap stays `gap-4`), each cell an `<a>` wrapping `<img src={Vutuv.Screenshot.url({url.screenshot, url}, :thumb)} width="400" height="264" loading="lazy" class="aspect-[400/264] w-full object-cover">` inside `overflow-hidden rounded-lg ring-1 ring-slate-200 dark:ring-slate-800`, then a `min-w-0` title/URL block (`truncate`). The screenshots come from `Vutuv.PageScreenshot` (headless Chromium, on link create/update); `Screenshot.url/2` falls back to `/images/screenshot.png`.

### CSS architecture (don't break it)

`app.css` order: `@import "tailwindcss"` ŌåÆ `@import "./components.css" layer(components)`,
**at the top** before `@source`/`@theme` ŌĆö the `@import ŌĆ” layer()` only works there
(CSS ignores `@import` after other rules). `components.css` lives in the `components`
layer, so it beats Preflight (base) but loses to Tailwind utilities, which lets
hand-written pages use utilities freely. There is **no `legacy.css` anymore** ŌĆö do not
reintroduce it (a regression test enforces this). Dev serves `/assets/app.css`
undigested, so hard-reload (Cmd+Shift+R) after a rebuild. **Dev does not use
`tailwind --watch`:** the v4 CLI's watch mode rebuilds from a cached copy of
`@import`ed CSS and ignores CSS edits outright (verified on 4.0.0 and 4.3.0), so
`VutuvWeb.TailwindWatcher` (wired as the dev watcher) runs a correct one-shot
build on every CSS/template/JS change instead. Edits to `components.css` reach
the browser within ~2s; if they ever don't, suspect that watcher first.

### Don'ts

- No theme toggle (dark = system).
- Don't reskin legacy pages per-template ŌĆö edit `components.css`.
- Don't invent new bespoke colours; stay on brand / accent / slate.
- No em-dashes in UI copy (project-wide rule).
